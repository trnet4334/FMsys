## Context

此變更為跨模組架構調整，涵蓋資料模型、快照作業流程、前端儀表板、報表與安全治理。現況問題是資料來源分散且粒度不一致，導致淨值與績效無法穩定回溯。設計上採「快照為唯一分析來源」，由交易系統提供持倉彙總與損益資訊，儀表板系統負責週/月快照固化、跨幣別估值、差異計算、視覺化與報表輸出。

主要限制包含：外部交易 API 規格尚待最終定版、匯率來源品質需可驗證、多租戶未在 MVP 一次到位但資料模型需預留隔離能力。主要利害關係人為主用戶（個人資產管理者）、家庭共用查看者、未來 SaaS 用戶與系統維運者。

## Goals / Non-Goals

**Goals:**
- 建立可重現的快照資料鏈路，讓同一時點淨值、配置與績效可一致回放。
- 在 Web MVP 提供可用的淨值儀表板與關鍵分析，並可逐期擴展到收支、報表與風控。
- 明確定義外部交易系統與本系統責任邊界，避免逐筆交易邏輯混入儀表板。
- 建立安全與稽核基線（認證、授權、加密、備份、審計）以支援多用戶演進。

**Non-Goals:**
- 不在本變更中實作交易下單、委託管理或撮合流程。
- 不在 MVP 階段完成 App 客戶端與所有第三方券商直連。
- 不在本變更中提供完整 AI 投資建議或進階量化風險模型。

## Decisions

1. 快照優先而非交易明細優先
- Decision: 以 `snapshots` + `snapshot_holdings` + `snapshot_diffs` 作為分析主資料層，報表與圖表僅讀取快照層。
- Rationale: 可穩定支援週/月回溯，並隔離外部交易系統的即時抖動。
- Alternatives considered:
  - 直接以逐筆交易即時計算：彈性高但計算與回補複雜，且與外部交易邊界衝突。
  - 僅存聚合值不存明細：查詢快但無法支撐下鑽與差異診斷。

2. 同步 + 非同步混合流程
- Decision: 觸發快照使用同步 API 發起，實際資料抓取與報表生成走佇列非同步，透過 webhook 回傳完成事件。
- Rationale: 降低前端等待時間並提高重試彈性。
- Alternatives considered:
  - 全同步：實作簡單但 timeout 與尖峰風險高。
  - 全事件驅動：彈性高但初期觀測與除錯成本較高。

3. 多幣別採「快照時固化匯率」
- Decision: 每次快照儲存完整匯率映射至 `exchange_rates`，估值結果同時保存原幣與主幣。
- Rationale: 保證歷史報表可重現，不受後續匯率更新影響。
- Alternatives considered:
  - 查詢時即時換算：最新性高但歷史不一致。
  - 只存單一主幣值：儲存較小但喪失匯損益拆解能力。

4. 能力分期交付與規格先行
- Decision: 以 capability 為單位分階段（P0-P3）交付，先完成核心可觀測的淨值與快照流程。
- Rationale: 可在最短時間驗證核心價值並降低一次性整合風險。
- Alternatives considered:
  - 一次建完整平台：需求完整但交付週期過長、風險集中。

5. 安全基線預設啟用
- Decision: 身份驗證採 JWT + OAuth2，MFA 與 RBAC 在架構層即納入，敏感資料加密與稽核日誌列為必備。
- Rationale: 財務資料屬高敏感，後補安全控制成本高且風險大。
- Alternatives considered:
  - 先單帳號後補安全：開發快但不符合產品定位與未來多用戶需求。

## Risks / Trade-offs

- [外部 API 規格變動] → 以 adapter 層封裝、建立契約測試與 mock server，降低上下游耦合。
- [快照任務失敗或延遲] → 佇列重試 + dead-letter queue + 作業監控面板。
- [匯率來源中斷或異常] → 多來源 fallback、快取 TTL、人工覆寫流程。
- [報表生成成本高] → 預先計算摘要、分層快取、限制大區間即時查詢。
- [多能力同時推進導致範圍膨脹] → 以 P0/P1 驗收門檻控管，不滿足門檻不進入下一階段。

## Migration Plan

1. 建立資料表與索引（snapshots/snapshot_holdings/snapshot_diffs/accounts）並部署到測試環境。
2. 上線快照觸發 API 與 webhook 回寫機制，先以手動觸發驗證端到端資料流。
3. 導入匯率快照與多幣別估值，與既有手動資料雙寫比對 1-2 週。
4. 切換儀表板資料來源至快照層，保留舊資料查詢為唯讀備援。
5. 啟用報表排程與警示通知，逐步打開 Email 通知開關。
6. 驗證安全與稽核策略後，關閉舊路徑寫入。

Rollback strategy:
- 保留舊版讀取 API 與資料快照，若新流程異常則將前端讀取切回舊來源。
- 關閉排程與 webhook 消費者，使用最後成功快照維持唯讀展示。

## Open Questions

- 外部交易系統 API 的認證模式、限流規則與錯誤碼契約是否已凍結。
- MVP 是否納入負債（房貸/信貸/信用卡）與保險資產，或延後至 Phase 3+。
- 報表模板是否需要客製區塊編輯器，或先採固定版面。
- 部署與維運每月預算上限，將影響 queue/reporting 架構規模。
- 合規範圍是否僅限台灣個資法，或需預先滿足跨區法規要求。
